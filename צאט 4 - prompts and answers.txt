Q: What are the benefits of splitting a Node.js project into multiple files?
   Give practical examples for beginners.

A: מעולה! בוא נדבר על היתרונות של פיצול פרויקט Node.js לכמה קבצים, עם דוגמאות מעשיות.

   הפרויקט שלך כאן הוא דוגמה מצוינת — בדיוק עשית את זה! יש לך index.js,
   courses.js ו-students.js — כל אחד עם תפקיד מוגדר.

   למה לפצל? 5 יתרונות מרכזיים:

   1. קריאות — למצוא דברים בקלות 🗂️
   אם כל הקוד נמצא בקובץ אחד של 500 שורות, כל שינוי קטן הופך לחיפוש ארכיאולוגי.
   בלי פיצול — היית צריך לגלול ולחפש איפה נמצאים הנתונים של הקורסים, איפה
   הפונקציה של בדיקת ת"ז... הכל מעורבב. עם פיצול — מיד יודעים: courses.js ←
   הגדרת מערך הקורסים, students.js ← בדיקת ת"ז + מערך הסטודנטים, index.js ←
   הגדרת השרת וה-Endpoints. לוחצים על הקובץ, ישר במקום.

   2. שימוש חוזר — לכתוב פעם אחת, להשתמש בכל מקום 🔄
   הפונקציה isValidIsraeliId מוגדרת פעם אחת ב-students.js. כל Route שצריך
   לאמת ת"ז — פשוט מייבא את הפונקציה. בלי להעתיק שוב את הקוד. export מצד אחד,
   import מצד שני — וזהו.

   3. אחריות מופרדת — כל קובץ "מתמחה" בדבר אחד 🎯
   courses.js — הנתונים של הקורסים. ורק הם. students.js — הנתונים של הסטודנטים
   + לוגיקת ת"ז. ורק הם. index.js — השרת, Routes, Middleware. ורק זה.
   אם מחר צריך לשנות את מבנה הקורס — נוגעים רק ב-courses.js. לא צריך לגעת
   ב-index.js. פחות סיכוי לשבור משהו.

   4. עבודה בצוות — כל אחד בקובץ שלו 👥
   אחד עובד על הוספת שדות לקורס ← courses.js. השני מוסיף Route חדש לסטודנטים ←
   index.js. אפס התנגשויות.

   5. דיבוג קל יותר 🐛
   אם יש באג בבדיקת ת"ז, אתה יודע בדיוק איפה לחפש: students.js. לא צריך לעבור
   על 200 שורות של Routes.

   איך זה עובד טכנית? המילים export ו-import הן המפתח:
   // קובץ A — מייצא
   export const myData = [1, 2, 3];        // named export
   export default myData;                   // default export
   // קובץ B — מייבא
   import myData from './a.js';             // default import
   import { myData } from './a.js';         // named import

   פיצול קבצים הוא לא "טכניקה מתקדמת" — זה הרגל בסיסי ונכון.

Q: Explain Node.js modules to a beginner.
   Show how one file exports data and another file imports it using import.

A: Module = קובץ JavaScript אחד. כל קובץ .js בפרויקט שלך הוא Module עצמאי.
   כברירת מחדל, כל מה שכתוב בתוך קובץ — משתנים, פונקציות, מערכים — פרטי לאותו
   קובץ. קבצים אחרים לא רואים אותו.

   כדי שקובץ B יוכל להשתמש במשהו מקובץ A, צריך שני דברים: קובץ A צריך לייצא
   (export) את מה שהוא רוצה לשתף, וקובץ B צריך לייבא (import) את מה שהוא צריך.

   דוגמה חיה מהפרויקט:
   צד א' — הייצוא (students.js):
   export function isValidIsraeliId(id) { ... }  ← named export
   export default studentsArr;                    ← default export

   צד ב' — הייבוא (index.js):
   import studentsArr, { isValidIsraeliId } from './students.js';
   //     ^^^^^^^^^^^    ^^^^^^^^^^^^^^^^^
   //     default        named
   //     בלי {}         חייב {}
   //     שם חופשי       שם נעול

   שני סוגי Export:
   Default — export default x, מייבאים בלי {}, אפשר לשנות שם, אחד לקובץ.
   Named — export function / export { x }, מייבאים עם {}, חייבים שם מדויק,
   ללא הגבלה.

   שים לב: חייבים "type": "module" ב-package.json כדי להשתמש ב-import/export.

Q: What does "one responsibility per file" mean in software development?
   Explain it using a beginner-friendly Express project.

A: "אחריות אחת לקובץ" — כל קובץ צריך לעשות דבר אחד, ולעשות אותו טוב. תחשוב על
   ארון בגדים מסודר: חולצות במגירה אחת, מכנסיים במגירה שנייה. כשאתה מחפש גרביים —
   אתה יודע בדיוק איזו מגירה לפתוח. ככה גם קוד.

   דוגמה חיה — הפרויקט שלך כבר מיישם את העיקרון:
   homeWork/
   ├── index.js      ← אחריות: השרת וה-Routes
   ├── courses.js    ← אחריות: נתוני הקורסים
   └── students.js   ← אחריות: נתוני הסטודנטים + תיקוף ת"ז

   courses.js — רק קורסים. הקובץ הזה לא יודע שיש שרת, לא יודע שיש סטודנטים,
   לא מטפל בבקשות HTTP. רק קורסים.
   students.js — רק סטודנטים. לא יודע על קורסים, לא מגדיר Routes.
   index.js — רק השרת. מתזמר — מחבר בין הנתונים לבין השרת. לא מגדיר נתונים.

   למה זה חשוב?
   1. קל למצוא דברים — פותחים את הקובץ הנכון, משנים, סוגרים.
   2. שינוי בקובץ אחד לא שובר קובץ אחר — משפרים את isValidIsraeliId בלי לגעת
      ב-Routes.
   3. עבודה בצוות — כל אחד עובד על הקובץ שלו, אפס התנגשויות.

   כלל אצבע: "האם אני יכול לתאר במילה אחת מה הקובץ הזה עושה?" — courses.js =
   קורסים. students.js = סטודנטים. index.js = שרת. אם התשובה היא "הוא עושה
   קורסים וגם סטודנטים וגם את השרת" — הגיע הזמן לפצל.

Q: What is usually the responsibility of app.js in an Express application?
   What kinds of code should stay in app.js and what should be moved elsewhere?

A: app.js הוא המנצח של התזמורת. הוא לא מנגן בעצמו — הוא מחבר בין כל הכלים ומוודא
   שכולם עובדים יחד.

   מה כן שייך ל-app.js? ✅
   1. יצירת האפליקציה והגדרות בסיסיות — const app = express(), app.use(express.json())
   2. חיבור ה-Routes — app.use('/courses', coursesRouter)
   3. Middleware גלובלי — 404 handler, Error handler (רשתות הביטחון)
   4. הרמת השרת — app.listen(port, ...)
   5. פונקציות עזר קטנות שקשורות לשרת — sendJson() עטיפה דקה סביב res.json

   מה לא שייך ל-app.js? ❌
   1. נתונים — שייכים ל-Data Layer (קבצים נפרדים)
   2. לוגיקה עסקית — שייכת ל-Service Layer (פונקציות חישוב, ולידציה)
   3. Routes של ישות ספציפית — כשהפרויקט גדל, עוברים ל-Express.Router בקובץ נפרד

   כלל הזהב:
   קשור ל-Express? ← ב-app.js
   עובד גם בלעדיו? ← בקובץ נפרד

   השאלה: "האם הקוד הזה קשור ישירות ל-Express, או שהוא היה עובד גם בלעדיו?"

Q: I didn't understand the two types of export.

A: המשל של החנות 🏪 — תחשוב על קובץ כמו חנות. כשהיא מייצאת מוצרים, יש שתי דרכים:

   Default Export — המוצר הכוכב ⭐
   לכל חנות יש מוצר אחד מרכזי. אצלך: export default coursesArr — "המוצר העיקרי
   של הקובץ הזה". כשמייבאים: import coursesArr from './courses.js' — אפשר לקרוא
   לזה איך שרוצים! import myCourses, import x — הכול יעבוד. השם ב-import לא חייב
   להתאים לשם המקורי. בגלל שיש רק אחד — אי אפשר להתבלבל.

   Named Export — מוצרים עם תווית שם 🏷️
   export { printCoursesToConsole } או export function isValidIsraeliId(id) {...}
   — השם ננעל. חייבים להשתמש בו בדיוק: import { isValidIsraeliId } from ...
   ✅ import { isValidIsraeliId } — עובד. ❌ import { checkId } — לא עובד.
   הסוגריים המסולסלים {} הם השלט "אני מחפש מוצר עם תווית שם ספציפית".

   שילוב:
   import studentsArr, { isValidIsraeliId } from './students.js';
   //     ^^^^^^^^^^^    ^^^^^^^^^^^^^^^^^
   //     default        named
   //     בלי {}         חייב {}
   //     שם חופשי       שם נעול

   Default — כשיש דבר אחד מרכזי: מערך הקורסים, מערך הסטודנטים.
   Named — כשיש כמה דברים: גם מערך (default) וגם פונקציית תיקוף ת"ז (named).
   Default = המוצר העיקרי. Named = מוצרים מהמדף עם ברקוד.

Q: Explain Express Router to a beginner.
   Why is Express Router useful in larger applications?
   Show the smallest possible Express Router example. Explain every line of code.
   Explain app.use() in Express.
   Show how to connect a router to an Express application.
   Explain what happens when a request arrives.

A: Router ב-Express הוא מפצל כבישים. במקום שכל הבקשות יגיעו לצומת אחד ענק,
   ה-Router מפנה כל בקשה לנתיב הנכון.

   בלי Router, app.js נראה ככה — קובץ אחד עם 50+ Routes, סיוט. עם Router —
   app.js נקי, כל ישות בקובץ משלה:
   app.use('/courses', coursesRouter);
   app.use('/students', studentsRouter);

   הדוגמה הכי קטנה שאפשר:
   // routes/courses.js
   import { Router } from 'express';   // (1) שליפת Router מ-Express
   const router = Router();            // (2) יצירת מופע Router נקי
   router.get('/', ...);               // (3) הגדרת Routes על ה-Router
   router.post('/', ...);
   export default router;              // (4) ייצוא

   // app.js
   import coursesRouter from './routes/courses.js';
   app.use('/courses', coursesRouter);  // (5) חיבור — "בקשות ל-/courses ← coursesRouter"

   app.use() — הפונקציה שמחברת Router ל-Express. הקידומת /courses נחתכת
   אוטומטית מה-URL לפני שה-Router רואה אותה. GET /courses/5 ← Router רואה
   GET /5.

   מה קורה כשבקשה נכנסת?
   GET /courses/5 ← Express בודק app.use('/courses', coursesRouter) ←
   הקידומת תואמת ← Express מעביר ל-coursesRouter ← Router מחפש route שמתאים
   ל-GET /5 ← מוצא router.get('/:id') ← מריץ את הפונקציה.

   סיכום: Router = שוטר תנועה. מחליט לאן כל בקשה הולכת.

Q: In a beginner Express project without a database,
   what is the responsibility of the data layer?
   Should Express routes be written inside the data layer? Explain why.

A: Data Layer זו השכבה שמחזיקה את הנתונים. בפרויקט בלי מסד נתונים, הנתונים
   יושבים במערכים פשוטים בתוך משתנים — בדיוק כמו בפרויקט שלך.

   המבנה:
   ┌─────────────────────────────┐
   │ index.js                    │ ← Routes Layer — מקבל בקשות HTTP, שולח תשובות
   ├─────────────────────────────┤
   │ courses.js — const arr      │ ← Data Layer — מחזיק נתונים, מייצא
   │ students.js — const arr     │
   └─────────────────────────────┘

   האחריות של Data Layer:
   1. להחזיק את הנתונים עצמם — const coursesArr = [...]
   2. לוגיקה שקשורה לנתונים — ורק להם — isValidIsraeliId, printCoursesToConsole
   3. לייצא את הנתונים החוצה — export default coursesArr

   האם Routes צריכים להיות בתוך Data Layer? ❌ ממש לא.
   למה? 3 סיבות:
   1. אי אפשר להשתמש בנתונים בלי Express — אם courses.js היה מכיל Router, אי אפשר
      היה להדפיס קורסים לקונסול בלי להרים שרת.
   2. בדיקות הופכות לסיוט — כדי לבדוק ת"ז צריך לדמות שרת שלם.
   3. שינוי קטן שובר דברים לא קשורים — מחליפים ספרייה? Data Layer ממשיך לעבוד.

   כלל פשוט: האם השורה מזכירה req, res, app, או router?
   כן ← Routes Layer. לא ← Data Layer.

Q: Explain the purpose of the service layer in an Express application.
   Show simple examples of functions that belong in a service.
   Do not write Express routes.

A: Service Layer היא השכבה שיושבת בין ה-Routes לנתונים. היא מכילה את הלוגיקה
   העסקית — החוקים של האפליקציה.

   בקשה ← Routes ← Service ← Data
           (HTTP)  (לוגיקה) (מערכים)

   כרגע בפרויקט שלך, הלוגיקה יושבת בתוך ה-Routes:
   app.post('/students', ...) {
       // ⬐ חוקים עסקיים — שייכים ל-Service
       if (!fullName) { ... }
       if (!isValidIsraeliId(id)) { ... }
       if (studentsArr.some(s => s.id === id)) { ... }
       // ⬏
   }

   איך נראה Service? דוגמה — יצירת סטודנט:
   export function createStudent({ fullName, birthDate, id }) {
       if (!fullName)  return { error: 'fullName is required' };
       if (!birthDate) return { error: 'birthDate is required' };
       if (!id)        return { error: 'id is required' };
       if (!isValidIsraeliId(id)) return { error: 'Invalid Israeli ID' };
       if (studentsArr.some(s => s.id === id))
           return { error: 'Student with this ID already exists' };

       const newStudent = { id, fullName, birthDate, courses: [] };
       studentsArr.push(newStudent);
       return newStudent;
   }

   דוגמה — עדכון קורס:
   export function updateCourse(id, { name, description }) {
       const course = coursesArr.find(c => c.id === id);
       if (!course) return { error: 'Course not found' };
       if (!name || name.trim() === '') return { error: 'Name is required' };

       course.name = name;
       course.description = description !== undefined ? description : course.description;
       return course;
   }

   שימו לב: הפונקציות לא רואות req/res. הן מקבלות ערכים פשוטים ומחזירות תוצאה או
   { error }. ה-Route יודע להמיר את זה ל-HTTP Status — ה-Service לא יודע כלום
   על HTTP.

   היתרון: הלוגיקה נבדקת בקלות בלי שרת. import { createStudent } from '...' —
   וזהו. אפשר גם להשתמש באותה לוגיקה ממקומות שונים (Route, CLI, Script וכו').

Q: What is the responsibility of a controller in an Express application?
   How is it different from a service?
   Give beginner-friendly examples.

A: Controller הוא מתווך. התפקיד היחיד שלו: לקחת בקשה מ-HTTP, להעביר אותה
   ל-Service, ולהחזיר תשובה ללקוח. הוא לא חושב, לא מחשב, לא בודק חוקים —
   רק מתרגם.

   Controller מול Service — מה ההבדל?
   Controller — יודע על HTTP. מקבל (req, res), קורא req.params / req.body,
   קורא ל-Service, שולח res.status().json(). דוגמה:
   export function create(req, res) {
       const result = createCourse(req.body);
       if (result.error) {
           return res.status(400).json(result);
       }
       res.status(201).json(result);
   }

   Service — לא יודע על HTTP. מקבל ערכים פשוטים, מחזיר תוצאה או { error }.
   דוגמה:
   export function createCourse({ name, description }) {
       if (!name || name.trim() === '') return { error: 'Name is required' };
       // ... יוצר קורס, מחזיר אותו
   }

   למה להפריד? אותו Service משרת Routes שונים — POST, PUT, PATCH — בלי לשכפל
   קוד. ה-Controller רק מתרגם את הקריאה ל-Service ומחזיר ללקוח.

   שלוש השכבות בפרויקט: Controller — "איך לענות?", Service — "מה החוקים?",
   Data — "איפה המידע?"

Q: What is the difference between an Express Router and a Controller?
   Use a simple CRUD example.

A: Router = שוטר תנועה. מחליט לאיזה Controller ללכת.
   Controller = פקיד קבלה. מקבל בקשה, מעביר ל-Service, מחזיר תשובה.

   דוגמה:
   // Router
   router.get('/', coursesController.list);        // GET    /courses
   router.get('/:id', coursesController.get);      // GET    /courses/5
   router.post('/', coursesController.create);     // POST   /courses
   router.put('/:id', coursesController.update);   // PUT    /courses/5
   router.delete('/:id', coursesController.remove);// DELETE /courses/5

   // Controller
   export function get(req, res) {
       const result = getCourse(req.params.id);
       if (result.error) return res.status(404).json(result);
       res.json(result);
   }

   Router: "בקשה ל-GET /courses/5? ← activate get בקונטרולר. בקשה ל-POST
   /courses? ← activate create." Controller: "הבקשה הגיעה אליי. אני שולף
   req.params.id, קורא ל-Service, מחזיר JSON."

   ההבדל המהותי: Router עוסק ב"לאן" — התאמת URL+Method לפונקציה. Controller
   עוסק ב"איך לענות" — תרגום HTTP ↔ Service. Router = שלט הכוונה, Controller =
   האדם שמולך בדלפק.

Q: Explain the responsibility of each layer.

A: POST /courses — Body: { "name": "שיעור תהילים", "description": "פעם בשבוע" }

   זרימת הבקשה:
   לקוח ← App ← Router ← Controller ← Service ← Data

   App: רושם Middleware, מחבר Routers. לא מעורב בבקשה עצמה — רק בהגדרות.
   Router: "בקשה POST /courses — ללכת ל-createController."
   Controller: שולף req.body, קורא ל-createCourse({name, description}).
       אם Service מחזיר { error } ← res.status(400).json.
       אם Service מחזיר קורס ← res.status(201).json.
   Service: בודק תקינות, יוצר ID, שומר, מחזיר קורס חדש.
   Data: מחזיק את המערך — studentsArr, coursesArr. לא יודע על HTTP.

Q: Why app.use('/courses', enrollmentRouter)?
   Explain to me why this line is needed.

A: This single route needs to exist somewhere:
   GET /courses/3/students ← "Who is enrolled in course #3?"

   Option A: Put it in coursesRouter ❌ — coursesRouter would then import from
   enrollmentController. That mixes responsibilities. The courses Router should
   only know about courses CRUD, not about enrollment. It violates "one
   responsibility per file."

   Option B: Create a separate Router ✅
   But to reach it, someone has to tell Express: "when you see /courses, also
   check this Router." That's: app.use('/courses', enrollmentRouter);

   Express allows multiple Routers on the same prefix. When a request arrives:
   GET /courses/3/students — Express checks in order:
   (1) coursesRouter — sees GET /3/students, no route matches (it only has
       /:id for GET single course, not /:id/students).
   (2) enrollmentRouter — sees GET /3/students, matches
       router.get('/:courseId/students').

   If enrollmentRouter were NOT registered, Express would skip it and fall through
   to the 404 handler. Same prefix, two different Routers, each with its own
   responsibility. app.use is not "one Router per prefix" — it's "add this
   Router to this prefix." You can stack them.

A: Found 4 issues in the code:

   Bug 1: updateCourse Controller — Missing name returns 404, should be 400
   Where: controllers/courses.js
   Problem: When the Service returns { error: 'Name is required' }, the
   Controller returned 404 (Not Found) instead of 400 (Bad Request). The
   Controller mapped every error to 404, but "name missing" is a bad request —
   not a resource not found.
   Fix: Added a check — if the error contains "not found" → 404, else → 400.

   Bug 2: unenrollStudent — 404 when "not enrolled", should be 409
   Where: controllers/enrollment.js
   Problem: When trying to unenroll from a course the student isn't enrolled
   in, the Controller returned 404. But 404 means "resource not found" — both
   the student and course exist, there's just no enrollment to remove. This is
   a conflict (409), not a not-found.
   Fix: Return 409 Conflict when the error is "not enrolled".

   Bug 3: enrollment.js Service — confusing field names in error objects
   Where: services/enrollment.js
   Problem: When enrollStudent couldn't find a course, the error was
   { error: 'Course not found', id: courseId }. "id" alone is unclear — whose
   id? Course? Student? Confusing.
   Fix: Changed to { error: 'Student not found', studentId } and
   { error: 'Course not found', courseId } — the field name reflects exactly
   what wasn't found.

   Bug 4: createCourse and updateCourse — empty name ("") passes validation in PUT
   Where: services/courses.js
   Problem: The Service checked if (!name). Empty string "" is falsy, so it's
   caught. But someone could send "   " (spaces) and that's truthy. Also, to be
   consistent with patchCourse.
   Fix: Changed to !name || name.trim() === ''— catches both "" and "   ".
