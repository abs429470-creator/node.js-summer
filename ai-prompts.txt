Q: I already know JavaScript for frontend development.
Explain the difference between frontend and backend development.
Focus on the responsibility of each side rather than specific technologies

A: Frontend runs in the user's browser and handles everything the user sees and
   interacts with — UI, UX, animations, local state, and initial input validation.
   Backend runs on a remote server and handles the real business logic, security
   and permissions, database access, and heavy processing. The frontend cannot be
   trusted because the user can modify it; the backend is the source of truth.
   Frontend is about how things look; backend is about what actually happens.

--------------------------------------------------------------------------------
PROMPT 2
--------------------------------------------------------------------------------
Q: Explain how a browser and a backend server communicate.
Describe the request-response cycle step by step.
Do not use code

A: The cycle has 9 steps: (1) User initiates an action in the browser.
   (2) Browser builds a structured request containing the HTTP method, URL,
   headers, and optional body. (3) The request travels over the internet through
   routers and DNS. (4) Server receives and parses the request. (5) Server
   processes it — runs business logic, accesses the database, checks permissions.
   (6) Server builds a response with a status code, headers, and body.
   (7) Response travels back to the browser. (8) Browser processes the response
   — checks the status code and updates the UI accordingly. (9) Browser waits
   for the next user action and the cycle begins again.

PROMPT 3
--------------------------------------------------------------------------------
Q: Explain what an API is.
Give two real-world examples of APIs that are not related to programming.
Then explain how the same idea applies to web development

A: An API is a contract that lets two programs communicate. Two real-world
   analogies: (1) A waiter in a restaurant — you order from a menu, the waiter
   relays it to the kitchen, and you get a plate back without needing to know
   how the kitchen works. (2) A TV remote — buttons expose a simple interface,
   the remote sends an infrared signal, and the TV executes without you needing
   to understand its internal electronics. In web development, the server exposes
   predefined endpoints, the client sends a request in an agreed format, and the
   server returns a response. The client doesn't need to know how the server is
   built internally.

PROMPT 4
--------------------------------------------------------------------------------
Q: Explain the difference between HTML and JSON.
When is each format typically returned by a web server?
Do not assume any specific framework

A: HTML contains both content and display instructions — it's a ready-to-render
   page for the human eye. JSON contains only raw structured data — it's meant
   for code to process. A server returns HTML when the user types a URL in the
   browser, when doing server-side rendering, or for static content pages.
   It returns JSON for background data requests, form submissions without page
   refresh, Single Page Applications where the backend is purely a data source,
   and mobile or non-browser clients.

--------------------------------------------------------------------------------
PROMPT 5
--------------------------------------------------------------------------------
Q: Can we manage without returning HTML at all?
Do not assume any specific framework

A: Yes. This is the Single Page Application (SPA) architecture. The server sends
   one minimal HTML shell, and JavaScript builds everything else. All subsequent
   communication is JSON only. Benefits: complete separation of concerns, one
   backend serves web/mobile/all clients, smooth user experience without page
   refreshes, and parallel frontend/backend development. Trade-offs: slower
   initial load, SEO challenges, and added complexity in state management,
   routing, and browser back-button handling.

--------------------------------------------------------------------------------
PROMPT 6
--------------------------------------------------------------------------------
Q: Explain more about the bottom line (regarding SPA vs traditional vs combined)

A: SPA — a single HTML page stays throughout the user's visit. JavaScript manages
   everything: when the user clicks a link, JavaScript stops the default behavior,
   requests JSON from the server, and updates only the relevant parts of the
   screen. Benefits: smooth UX, clear separation of concerns, simpler server.
   Drawbacks: SEO problems, heavy initial load, manual browser history management,
   complex client-side state. When to use SPA: interactive apps like dashboards,
   tools, email clients. When not: content sites, blogs, e-commerce product pages.
   The combined approach (SSR + Hydration) gives fast initial load and SEO
   from traditional, plus smooth navigation from SPA — but doubles complexity
   since components must run on both server and browser.

--------------------------------------------------------------------------------
PROMPT 7
--------------------------------------------------------------------------------
Q: What did you mean by "it gives the benefits of both approaches — and the
   complexity of both"?

A: The combined approach (SSR + Hydration) gets fast first load and SEO from
   the traditional approach, plus smooth transitions and rich interactivity from
   SPA. But it also inherits the complexity of both: the server must render HTML
   (traditional complexity), and the client must manage state and routing (SPA
   complexity). Additionally, the same component must run in both Node.js and
   the browser — two different environments. A worst-case scenario: the server
   sends HTML with price "100", but during load the price updates to "90" in the
   database. When JavaScript takes over and fetches fresh JSON, there's a
   mismatch between the displayed HTML and the app's state. Debugging across
   both environments is harder than debugging just one.

PROMPT 8
--------------------------------------------------------------------------------
Q: Explain why Express was created.
Compare creating a simple server with Node's http module and with Express.
Focus on readability and developer productivity

A: Express wraps Node's built-in http module with a convenience layer. With http,
   you write one large function with if-else chains to check URLs and methods,
   manually parse the body, manually set every header, and manually stringify
   JSON. With Express, you define routes cleanly (method + path + handler),
   Express handles routing, body parsing, automatic headers, and JSON
   serialization. Express didn't invent new capabilities — it packaged common
   patterns developers were rewriting in every project. The result: less
   boilerplate, better readability, and a modular structure that scales.

--------------------------------------------------------------------------------
PROMPT 9
--------------------------------------------------------------------------------
Q: Explain what a route is in Express.
Show three examples of GET routes and explain what each one could return.
Do not use POST, PUT or DELETE yet

A: A route in Express is a combination of an HTTP method, a URL path, and a
   handler function. Express checks incoming requests against defined routes in
   order — the first match runs. Three GET examples:
   (1) /products — returns a JSON array of all products (a collection).
   (2) /products/:id — returns a single product by ID, where :id is a dynamic
       parameter Express extracts automatically. Returns 404 if not found.
   (3) /dashboard/summary — returns computed/aggregated data from multiple
       sources, not just a raw database read.
   A route is also self-documenting — looking at a routes file reveals the
   entire API surface at a glance.

--------------------------------------------------------------------------------
PROMPT 10
--------------------------------------------------------------------------------
Q: Explain more about routes. Stick to the previous instructions.

A: A route is a routing system, not just "if URL is X, do Y." In a real app
   there are tens of routes, and Express matches them by order of definition.
   The route structure: method (GET is safe and cacheable — must not change
   server state), path (three types: fixed like /about, parameterized like
   /products/:id, and wildcard with *), and handler(s) (can be a chain of
   functions passing control via next() — middleware). Express finds routes by
   iterating in definition order; the first match runs and the rest are skipped.
   Response types beyond JSON: HTML, XML, plain text, file downloads, redirects,
   and streams for large content. If no route matches, Express does nothing
   unless a catch-all returns 404.

PROMPT 11
--------------------------------------------------------------------------------
Q: In Node.js, what is a module?
Explain how one JavaScript file can use code from another JavaScript file.
Show a very small example using module.exports and require

A: A module in Node.js is simply a JavaScript file. Variables defined in one
   file are not visible in another unless explicitly exported. The exporting file
   uses module.exports to expose functions/objects. The importing file uses
   require('./path') to receive them. Example: math.js exports { add, subtract },
   and app.js imports them via const math = require('./math.js').
   This modular system allows splitting code into small, focused files — without
   it every Node.js project would tend toward one giant file.

PROMPT 15
--------------------------------------------------------------------------------
Q: Explain the difference between CommonJS and ES Modules. Explain the problem.

A: CommonJS (require/module.exports) is the original Node.js module system — it's
   synchronous, require can appear anywhere in code (dynamic), and what
   module.exports returns is exactly what require receives. ES Modules
   (import/export) is the JavaScript standard — it's asynchronous by nature
   (browsers need to download files), import must be at the top of the file
   (static), and it distinguishes between export default and named exports.
   The problem: index.js uses ES Modules (import), but courses.js used
   module.exports (CommonJS). Node.js cannot resolve module.exports as a default
   import, hence the error. Fix: use export default in courses.js and students.js
   to match the ES Modules system index.js already uses.

Q: How will the server support multiple different routes? Give me 4 examples with
   an explanation of each line of code.

A: With Node's http module, you check req.url and req.method manually in an
   if-else chain. Four examples:
   (1) if (req.url === '/' || req.url === '/home') — returns a homepage message
       with status 200 and text/plain content type.
   (2) else if (req.url.startsWith('/courses/')) — splits the URL to extract a
       dynamic ID via req.url.split('/')[2] and returns a response with that ID.
   (3) else if (req.url === '/api/courses') — returns JSON.stringify(coursesArr)
       with Content-Type: application/json.
   (4) else — returns 404 with "page not found."
   This approach works but becomes unwieldy as routes grow — Express solves this.

PROMPT 18
--------------------------------------------------------------------------------
Q: Explain the Express code to me (rewriting the current server in Express).

A: import express from 'express' — imports Express as a default export.
   const app = express() — creates the central app object; everything runs
   through app. app.get('/', handler) — defines a GET route for the root path.
   res.send(string) — sends a response; Express automatically sets Content-Type
   to text/html. /courses/:id — Express extracts dynamic parameters into
   req.params.id, no manual string splitting. res.json(data) — sends JSON;
   Express automatically stringifies and sets Content-Type: application/json.
   app.listen(port, callback) — replaces http.createServer + server.listen.
   Express also provides automatic 404 handling when no route matches.
