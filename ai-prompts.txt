Q: Explain what CRUD means in backend development.
   Explain why almost every backend application needs CRUD operations.
   Explain CRUD using a course management system. do not write code

A: CRUD הן ארבע פעולות בסיסיות: Create (יצירה), Read (קריאה), Update (עדכון),
   Delete (מחיקה). בפיתוח backend הן ממופות ל-HTTP methods:
   Create=POST, Read=GET, Update=PUT/PATCH, Delete=DELETE.

   כמעט כל אפליקציה היא, בבסיס שלה, ניהול של מידע. המהות של תוכנה זה מידע — מה
   שאתה שומר, איך אתה מציג אותו, ואיך אתה משנה אותו. לקוחות, מוצרים, הודעות,
   קבצים, הגדרות… הכול מידע שעובר את אותו מסלול חיים: משהו נוצר, מוצג למישהו,
   משתנה כשצריך, ובסוף נמחק. הכול מתכנס ל-CRUD. גם שירות מורכב כמו מערכת
   תשלומים זה CRUD: Create לחיוב, Read לעסקאות, Update ליתרה, Delete לחיוב
   שבוטל. אפליקציית צ'אט: Create להודעה חדשה, Read לקריאת השיחה, Update
   לעריכת הודעה, Delete למחיקה. CRUD זו השפה המשותפת. מפתחים וצוותים מדברים
   במונחים האלה. כשמישהו אומר "ה-API תומך ב-CRUD מלא על קורסים" — כולם מבינים
   מיד אילו פעולות אפשר לבצע.

   דוגמה: מערכת ניהול קורסים בישיבה. Create — המזכיר מכניס שיעור חדש למערכת,
   ממלא שם קורס, מגיד שיעור, שעה ובית מדרש. ה-backend מקבל את הנתונים, מוודא
   שהם תקינים, ושומר. Read — בחור נכנס לרשימת הקורסים ורוצה לראות אילו שיעורים
   יש בזמן הזה, או פרטים על קורס ספציפי: מי מלמד, איפה. Update — המגיד שיעור
   עבר לבית מדרש אחר, המזכיר משנה את המיקום ושומר. Delete — קורס בוטל מחוסר
   נרשמים, המזכיר מוחק. CRUD הוא לא טכנולוגיה מסוימת או ספרייה — זאת פשוט הדרך
   הכי פשוטה ומדויקת לתאר מה ששרתים ובסיסי נתונים עושים.

Q: Explain REST APIs to a beginner.
   Focus on why HTTP methods are used instead of action names in URLs

A: REST API זו דרך לבנות שרת אינטרנט שמאפשר לתוכנות אחרות לתקשר איתו. במקום
   להחזיר דפי HTML (כמו אתר רגיל), השרת מחזיר מידע "נקי" — בדרך כלל ב-JSON.

   למה לא לשים שמות פעולה ב-URL? גישה בלי REST נראית כך:
   GET /getAllCourses, GET /createCourse?name=math, GET /deleteCourse?id=5 —
   כל פעולה צריכה URL משלה והרשימה מתארכת מהר. אין סטנדרט — כל מתכנת ממציא
   שמות משלו. קשה לנחש איך לבצע פעולה מסוימת.

   הפתרון: HTTP Methods + URL אחיד. ה-URL מייצג את המשאב (Resource), וסוג
   הפעולה נקבע ע"י ה-HTTP Method. אותו URL (/courses/5) משרת שלוש פעולות
   שונות — GET מחזיר, PUT מעדכן, DELETE מוחק. למה זה טוב? פשטות — במקום 20
   URLs למערכת יש 5-6. סטנדרט עולמי — כל מי שמכיר REST יודע מיד איך לעבוד.
   התאמה טבעית לפעולות CRUD. הפרדת אחריות — URL = מה, method = מה לעשות איתו.

PROMPT 5
--------------------------------------------------------------------------------
Q: Explain more about proper API design in REST API without writing code

A: 10 עקרונות בעיצוב REST API:

   1. שמות עצם, לא פעלים — URL מייצג משאב (דבר), לא פעולה. ❌ /getBooks ← ✅ GET /books

   2. רבים, לא יחיד — שמות אוספים תמיד ברבים. ❌ /book ← ✅ /books
      חריג: משאבים יחידים באופיים כמו /profile, /settings.

   3. קינון — עד רמה אחת, מקסימום שתיים. ❌ /books/5/reviews/3/comments/7 ←
      ✅ /books/5/reviews או /reviews/3/comments. כלל אצבע: אם אפשר לגשת
      למשאב ישירות, אל תכריח מסלול מקונן.

   4. מקפים (kebab-case) — ❌ /book_categories ← ✅ /book-categories

   5. גרסאות — API חייב להתפתח בלי לשבור לקוחות קיימים. שלוש גישות: ב-URL
      (/v1/books) — הכי פשוט ומומלץ למתחילים; ב-Header — URL נקי אבל פחות נוח;
      Query string — הכי קל למימוש אבל מלכלך. המלצה: תמיד תתחיל עם /v1/.

   6. סינון, מיון ופאג'ינציה — Query Parameters. GET /books?genre=fiction
      &sort=title&order=asc&page=2&limit=20. סינון: ?genre=torani. מיון:
      ?sort=title. כיוון: ?order=asc / desc. פאג'ינציה: ?page=2&limit=20.

   7. קודי סטטוס — 200 (הצלחה), 201 (נוצר), 204 (הצלחה בלי תוכן), 400 (בקשה
      לא תקינה), 401 (לא מחובר), 404 (לא נמצא), 409 (התנגשות), 422 (מידע לא
      תקין), 500 (שגיאת שרת).

   8. JSON עקבי — פריט בודד: {id, title...}. רשימה תמיד עטופה: {data: [...],
      total, page, limit}. מעטפת data מאפשרת להוסיף metadata בלי לשבור לקוחות.
      שגיאה תמיד במבנה אחיד: {error: {code, message, details}}.

   9. שמות שדות — camelCase: {publishYear, pageCount}. לא publish_year.

   10. POST מול PUT — POST יוצר (לא idempotent, אין ID), PUT מעדכן מלא
       (idempotent, חייב ID), PATCH עדכון חלקי.

   11. HATEOAS (אופציונלי) — כל תשובה מכילה קישורים לפעולות אפשריות: links: [
       {rel: "self", href: "/books/5"}, {rel: "author", href: "/authors/12"} ].
       כמעט אף אחד לא מממש, אבל כדאי להכיר.

   סיכום: URL = שם עצם ברבים, אל תקנן עמוק, Query string לסינון/מיון/פאג'ינציה,
   החזר JSON עקבי, השתמש בקודי סטטוס נכונים, תכנן גרסאות מראש.

Q: In Express, explain req.params and req.body.
   Describe when each one is used.
   Do not explain query parameters

A: req.params — נתונים מה-URL, משמש כשהערך הוא חלק מהכתובת עצמה, בדרך כלל
   לזיהוי משאב ספציפי. מוגדר עם :param ב-route: app.get('/books/:id', ...),
   נגיש כ-req.params.id (תמיד מחרוזת). מתי? כשרק צריך לזהות איזה משאב — שליפה,
   עדכון, מחיקה לפי ID. לא חייב express.json().

   req.body — נתונים מגוף הבקשה, משמש כשהלקוח שולח מידע ליצירה או עדכון, והמידע
   מורכב או גדול מדי בשביל URL. נגיש כ-req.body.field, דורש middleware של
   express.json(). מתי? כשיוצרים (POST) או מעדכנים (PUT) משאב עם נתונים שהלקוח
   מספק.

   שימוש משולב: PUT /books/:id — req.params.id אומר איזה ספר ("5"), req.body
   אומר מה לשנות ({title: "שם חדש"}).
