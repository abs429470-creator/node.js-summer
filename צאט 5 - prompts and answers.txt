1.
Why should a backend API return different HTTP status codes instead of always returning 200?
Explain the purpose of HTTP status codes. Focus on 200, 201, 401 and 404.

Answer:
בקיצור — **סטטוס קוד מספק שפה משותפת בין השרת ללקוח**. במקום שהלקוח יצטרך לנתח את גוף התשובה כדי להבין מה קרה, הוא מקבל מיידית אינדיקציה ברורה על תוצאת הבקשה.

אם תמיד מחזירים 200, הלקוח חייב:
1. לקרוא את גוף התשובה
2. להבין את המבנה הפנימי של השגיאות שלך
3. לכתוב לוגיקה ייעודית לכל מצב

זה שביר, לא סטנדרטי, ומקשה על כלים אוטומטיים (מוניטורינג, פרוקסי, caching) לעבוד נכון.

| קוד | משמעות | שימוש טיפוסי |
|-----|--------|---------------|
| **200 OK** | הבקשה הצליחה, והתשובה מכילה את מה שביקשת | `GET /users/5` — קיבלת את המשתמש |
| **201 Created** | המשאב נוצר בהצלחה | `POST /users` — משתמש חדש נוצר, לרוב עם כותרת `Location` שמפנה למשאב החדש |
| **401 Unauthorized** | הלקוח לא מזוהה — חסר טוקן או שהטוקן לא תקין | ניסיון לגשת ל-`/admin` בלי להיכנס קודם |
| **404 Not Found** | המשאב המבוקש לא קיים | `GET /users/999` — אין משתמש עם המזהה הזה |

בפרויקט coursesRouter.js אפשר לראות את היישום בפועל.

היתרון ברור: צד הלקוח יכול לכתוב לוגיקה פשוטה — `if (response.status === 404)` ומשם להתנהג בהתאם, בלי לנתח את הטקסט של השגיאה.

2.
In an Express application, when should I return HTTP 404?
Show a simple example using GET /courses/:id.
Explain why returning 404 is better than returning an empty response.

Answer:
מחזירים **404** כשמשאב ספציפי לא נמצא במערכת. המקרה הקלאסי: בקשת `GET` לפי מזהה, והמזהה לא קיים.

const courses = [
  { id: 1, name: 'מבוא לתכנות' },
  { id: 2, name: 'מבני נתונים' },
];

app.get('/courses/:id', (req, res) => {
  const course = courses.find(c => c.id === Number(req.params.id));
  if (!course) {
    return res.status(404).json({ message: 'הקורס המבוקש לא נמצא' });
  }
  res.json(course);
});

שלוש סיבות למה 404 עדיף על 200 עם null:
1. **הבחנה בין מצבים** — 200 עם `null` יכול לנבוע מקורס שלא קיים או ערך לגיטימי `null`. 404 מסיר עמימות.
2. **סמנטיקה לרשת** — דפדפנים, פרוקסי, וכלי caching מבינים 404. 200 עם `null` עלול להישמר במטמון.
3. **קוד לקוח פשוט יותר** — טיפול אחיד בשגיאות: `if (!response.ok)` תופס 404, 401, 500.

3.
What are HTTP request headers?
Explain why servers sometimes check request headers before processing a request. Do not explain JWT or OAuth.
Explain HTTP 401 Unauthorized.
Show a simple Express example that checks a custom request header.
Do not use JWT or authentication libraries.

Answer:
הדרים (headers) הם מטה-דאטה שנשלח יחד עם בקשת ה-HTTP — זוגות של מפתח-ערך. דוגמאות: Content-Type, Accept, Accept-Language, User-Agent, X-API-Key.

למה השרת בודק הדרים לפני עיבוד:
- **דחייה מהירה** — חוסך משאבים מיותרים
- **ניתוב נכון** — Accept קובע JSON/HTML/XML
- **הגנה בסיסית** — API key לפני גישה לנתונים

**401 Unauthorized** — "אתה לא מזוהה". שונה מ-403 (יודע מי אתה אבל חסר הרשאה).

const express = require('express');
const app = express();
const VALID_API_KEY = 'my-secret-key-123';

app.use('/admin', (req, res, next) => {
  const apiKey = req.headers['x-api-key'];
  if (!apiKey) return res.status(401).json({ message: 'חסר מפתח API' });
  if (apiKey !== VALID_API_KEY) return res.status(401).json({ message: 'מפתח API שגוי' });
  next();
});

app.get('/admin/stats', (req, res) => {
  res.json({ users: 150, courses: 12 });
});

Answer:
**HTTP Header** — שורת מטה-דאטה מחוץ לגוף הבקשה, מורכבת משם וערך (למשל `x-api-key: abc123`).

**סטטוס בקריאה לא מאומתת:** 401 Unauthorized.

**בדיקת הדרים בצד שרת (אקספרס):**
req.headers['x-api-key'] — אובייקט JavaScript רגיל. כל ההדרים זמינים ב-req.headers באותיות קטנות.

6.
Why is duplicating the same code in many Express routes considered bad practice?
Explain using request validation as an example.
What is the solution to prevent duplication the same code of validation?

Answer:
**שלוש בעיות:** (1) תחזוקה קשה — שינוי דורש עדכון בהרבה מקומות, (2) באגים שקטים — קל לשכוח מקום אחד, (3) בלגן — קוד אימות מתערבב בלוגיקה עסקית.

**הפתרון — מידלוור (Middleware):** פונקציה שרצה **לפני** ההנדלר של הראוט. מגדירים פעם אחת, מחילים על כל הראוטים שצריכים אימות:

function checkApiKey(req, res, next) {
  const apiKey = req.headers['x-api-key'];
  if (!apiKey) return res.status(401).json({ message: 'חסר מפתח' });
  if (apiKey !== 'secret-123') return res.status(401).json({ message: 'מפתח שגוי' });
  next();
}

app.use('/courses', checkApiKey);
app.use('/admin', checkApiKey);

זה כמו לשים מאבטח בכניסה לבניין במקום לבדוק תעודות בכל קומה.

8.
in node.js, Explain middleware in Express to a beginner. Use API key validation as the example. Explain what next() does.
Show how to create a simple Express middleware that checks a custom x-api-key header. Explain every line of code.

Answer:
**מידלוור** — פונקציה שיושבת באמצע בין הבקשה לראוט. כמו מאבטח בכניסה לבניין.

**next()** — האות "הכול בסדר, תמשיך". שלוש אפשרויות:
- `next()` — להמשיך למידלוור/ראוט הבא
- `return res.status(401).json(...)` — לעצור עם שגיאה
- `next(error)` — להעביר שגיאה לטיפול

הדוגמה המלאה בפרומפט 7, עם הסבר על כל פרמטר:
- **req** — אובייקט הבקשה (headers, body, params, query)
- **res** — אובייקט התשובה (json(), status(), send())
- **next** — פונקציה להעברה הלאה

החוק החשוב: תמיד לקרוא ל-next() או לשלוח תשובה. אחרת הבקשה תיתקע.

```javascript
const existingIds = new Set(coursesArr.map(c => Number(c.id)));
let newId = 1;
while (existingIds.has(newId)) newId++;
```
מה האלגוריתם עושה:
